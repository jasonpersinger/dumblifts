I Built a Workout Tracker in a Single HTML File

No build step. No framework. No node_modules. Just one file and a service worker.

Why One File

I wanted a workout tracker for my dumbbell routine. Every app I found wanted me to create an account, pick from 500 programs, or pay $10 a month. I just wanted to tap my sets and go.

So I built one. And I decided to keep it simple: everything in a single index.html file. HTML structure, CSS styles, and JavaScript logic all together. No build step, no bundler, no framework.

The entire app deploys by dragging a folder onto Netlify. The total file count is five: index.html, sw.js, manifest.json, and two icon PNGs. That's the whole thing.

The Stack

It's simpler than you'd expect for a fully functional PWA:

  • HTML + inline CSS + inline JS in one file (~2000 lines)
  • localStorage for data persistence (no database needed for single-device use)
  • Firebase for optional Google sign-in and cloud sync across devices
  • Service worker for offline support and installability
  • Google Fonts for Oswald (headings) and Source Sans 3 (body text)

No React. No Vue. No Svelte. No Tailwind. No npm install. No webpack. No vite. No turbopack. Just the browser APIs that have been there all along.

What It Does

More than you might expect from 2000 lines of vanilla code:

  • Workout tracking with tap-to-complete sets and per-set rep counting
  • Weight progression with +/- buttons in 5-pound increments
  • Automatic A/B alternation between workouts
  • Personal record detection that tells you when you hit a new max
  • Progress charts rendered as inline SVG (no charting library)
  • Custom workout builder with an exercise library of 20 movements
  • Rest timer with an animated countdown bar
  • Data export/import as JSON backup or CSV for spreadsheets
  • Cloud sync via Firebase for cross-device access
  • Full offline support through the service worker
  • Installable as a PWA on iOS and Android

The Tradeoffs

Let's be honest about what you give up:

No component model. Everything is a function that builds an HTML string and sets innerHTML. There's no virtual DOM, no reactive state, no lifecycle hooks. When state changes, the whole section re-renders. For a small app this is fine. For a large app it would be painful.

Global scope everything. There are 30+ functions and a handful of mutable global variables. Any browser extension or third-party script could collide with them. In practice this hasn't been a problem, but it's not how you'd build something bigger.

No tests. There's no test framework. The app is small enough that manual testing covers it. If I were adding complex features, I'd eventually want tests. But for a focused tool, shipping fast and testing manually has been the right call.

CSS gets long. About 900 lines of inline styles. They're organized with comments and CSS variables, but there's no scoping or modules. Everything is global CSS.

Would I Do It Again

Yes. Without hesitation.

Not everything needs a framework. Not everything needs a build step. For a focused tool with a clear purpose, a single file is the right architecture. You can read the entire codebase in one sitting. You can deploy it anywhere that serves static files. You can edit it in any text editor.

The moment you add a build step, you add complexity: dependency management, version conflicts, CI/CD pipelines, node version compatibility. For a personal project that does one thing well, that complexity isn't worth it.

The line I'd draw is around 3000 to 4000 lines. Beyond that, I'd start splitting into separate files (still no build step, just <script src> tags). Beyond maybe 8000 lines, I'd want a real framework. But most focused tools never get there.

Try It

DUMBLIFTS is live and free. The code is on GitHub if you want to look at how it works or fork it for your own project.
